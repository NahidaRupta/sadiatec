# ADR-002: Monorepo Structure and Upgrade Strategy

**Status:** Accepted  
**Date:** 2026-05-21  
**Authors:** Saidatech / techMirai

---

## Context

The template-clone model (ADR-001) gives each client a fully isolated deployment, but it
creates a maintenance problem: when we fix a bug or add a feature in shared code, we must
propagate that change to every client app without breaking any of them.

We need:
1. A clear boundary between shared code and per-client code
2. A repeatable, type-safe upgrade workflow
3. A reliable rollback procedure that does not require database migrations to reverse

---

## Decision

**Monorepo using pnpm workspaces with two zones:**

```
/
├── packages/
│   └── cms-core/          # shared, versioned, published internally
└── apps/
    ├── saidatech/         # Client: Saidatech HR/staffing
    ├── clinic-template/   # Client: Dental/medical clinic
    └── school-template/   # Client: School/daycare
```

Each `apps/<client>` depends on `cms-core` as a workspace package. `cms-core` is
versioned with semver and follows conventional commits to generate its CHANGELOG.

---

## Monorepo Structure Detail

### `packages/cms-core/`

Contains everything that is shared and should not be duplicated per client:

| Path | Contents |
|------|---------|
| `src/payload/config-factory.ts` | `buildCmsConfig(siteConfig)` — returns a Payload `Config` object merging base + client overrides |
| `src/payload/collections/` | Pages, Media, Settings, FAQs, News, Events, Gallery, Downloads, Team, Testimonials, Jobs — all with `aiVisible` field |
| `src/payload/blocks/` | Every block: four files each (`config.ts`, `component.tsx`, `adapter.ts`, `types.ts`) |
| `src/i18n/routing-factory.ts` | `buildRouting(siteConfig)` — returns a `defineRouting` config for next-intl |
| `src/i18n/messages/` | Base translation message files for en/ja/bn |
| `src/schema/site-config.ts` | `SiteConfig` Zod schema + TypeScript type (single source of truth) |
| `src/seed/` | Seed utilities and base seed data |
| `src/types/` | Shared TypeScript types not generated by Payload |

### `apps/<client>/`

Contains only what is unique to this client:

| Path | Contents |
|------|---------|
| `site.config.ts` | Client's `SiteConfig` instance — the only required file beyond `/public/brand` |
| `seed/` | Client-specific seed documents (homepage content, team members, initial FAQs, etc.) |
| `public/brand/` | logo.svg, favicon.ico, og-image.jpg, any brand-specific static assets |
| `collections/` | Client-exclusive collections not in `cms-core` (optional; rare) |
| `blocks/` | Client-exclusive blocks not in `cms-core` (optional; rare) |
| `messages/` | Client overrides to base translation messages (merge with cms-core base) |
| `payload.config.ts` | Thin wrapper: `import { buildCmsConfig } from 'cms-core'; export default buildCmsConfig(siteConfig)` |
| `next.config.ts` | Standard Next.js config, imports nothing from cms-core schema |

### What Lives Where (Rule of Three)

Before adding anything to an `apps/<client>` directory, ask:
- Will the second vertical need this? → Put it in `cms-core`
- Is this specific to this client's brand or content? → Put it in `site.config.ts` or seed
- Is it a visual asset? → Put it in `/public/brand`

If you add a client-exclusive collection, it must live in `apps/<client>/collections/` and
register itself via the `buildCmsConfig` overrides, not by modifying the factory.

## Upgrade Workflow

When `cms-core` receives a new feature, bug fix, or breaking change:

### Step 1: Prepare cms-core release

```bash
# In packages/cms-core/
pnpm changeset          # describe the change, mark as patch/minor/major
pnpm version-packages   # bumps package.json version, updates CHANGELOG.md
```

The CHANGELOG entry for every breaking change MUST include:
- What changed and why
- Migration steps for client apps (field renames, new required SiteConfig keys, etc.)

### Step 2: Upgrade a client app

```bash
# In apps/<client>/package.json:
# "cms-core": "workspace:^X.Y.Z"  →  "cms-core": "workspace:^X.Y.(Z+1)"
pnpm install
```

### Step 3: Type-check first

```bash
pnpm --filter <client> typecheck
```

TypeScript strict mode is the primary safety net. Any breaking change in a collection's
field types, the `SiteConfig` Zod schema, or a block's `types.ts` will surface here as a
compile error. Fix all type errors before proceeding.

### Step 4: Run smoke tests

```bash
pnpm --filter <client> test:smoke
```

Smoke tests cover:
- Payload admin can log in and list all enabled collections
- All enabled-feature pages return HTTP 200
- `/api/ai/corpus?locale=<default>` returns valid JSON (even when stubbed)
- Contact form submission reaches Resend sandbox without error

### Step 5: Deploy to preview

```bash
vercel --project <client> --env preview
```

Validate the preview URL against the critical user flows checklist defined per client in
`apps/<client>/e2e/`. Use the e2e-runner agent for systematic validation.

### Step 6: Promote to production

```bash
vercel --project <client> --prod
```

Only after preview validation passes. Do not batch multiple clients into one step.

### Step 7: Repeat per client

Run Steps 2–6 for each `apps/<client>` independently. If Step 3 fails for Clinic,
Saidatech can still proceed on its own schedule. Isolation is intentional.

## Rollback Steps

### Scenario A: Rollback before database migration (safest)

If the cms-core upgrade did not add new collections or fields that were already migrated
to production:

```bash
# In apps/<client>/package.json
# Revert version pin back to previous
pnpm install
vercel --project <client> --prod
```

No database action required. If the Vercel deployment fails health checks, `vercel rollback`
promotes the previous successful build instantly without any database interaction.

### Scenario B: Rollback after database migration

If the new cms-core version added Payload collections and the Payload migration already
ran on production:

1. **Do not drop the new columns/tables.** The previous cms-core version ignores columns
   it does not know about — Postgres schema additions are backward-compatible in this
   direction.
2. Revert the `cms-core` version pin and redeploy (same as Scenario A).
3. The old code runs correctly against the migrated schema; extra columns are invisible.
4. Log a follow-up task to clean up orphaned schema in a maintenance window, coordinated
   with a proper down-migration authored alongside the original up-migration.

### Scenario C: Emergency rollback via Vercel

```bash
vercel rollback --project <client>
```

Instantly promotes the previous successful deployment. The database is unaffected.
Follow Scenario B for schema cleanup if a migration had already run.

## Version Pinning Policy

- `apps/<client>/package.json` always pins an exact minor: `"workspace:^X.Y.Z"` — never `"*"` or `"latest"`
- `cms-core` CHANGELOG must exist and be accurate before a version is released
- **Patch versions** (X.Y.Z+1): bug fixes, backward-compatible. Clients should upgrade promptly.
- **Minor versions** (X.Y+1.0): new features, backward-compatible. Clients upgrade at their convenience.
- **Major versions** (X+1.0.0): breaking changes. All active client apps must be upgraded within 30 days.

---

## Consequences (Enables)

- A single `cms-core` bug fix propagates to all clients via a version bump, not copy-paste
- The type system enforces upgrade correctness — a new required `SiteConfig` field causes
  every client to fail at compile time (not runtime) until they provide it
- Clients with deployment freezes can stay on older `cms-core` versions without blocking
  other clients
- The CHANGELOG gives each client a clear audit trail of what changed and when

## Consequences (Constrains)

- pnpm workspaces is the required package manager for this monorepo — npm and yarn are not supported
- Adding a required field to `SiteConfig` is a major version bump, not minor
- Client-exclusive collections in `apps/<client>/collections/` are invisible to `cms-core`
  logic (including the AI corpus endpoint) unless explicitly registered via `buildCmsConfig` overrides
- The `apps/<client>` directories share a monorepo but have completely separate Vercel projects.
  There is no "deploy all" command — isolation is intentional.

## Open Questions

1. **Two locations with different booking systems:** If a Clinic client has two locations
   each using a different booking provider, the current `integrations.bookingProviderUrl`
   scalar is insufficient. A `locations[]` array with per-location booking config may be
   needed in a future SiteConfig minor version. See variability-matrix.md row 46.

2. **cms-core publication strategy:** Currently workspace-internal. If we ever want to
   share `cms-core` across external agencies or open-source it, it will need publishing to
   a private npm registry (e.g., GitHub Packages). Deferred until a fourth client.

3. **Shared translation base merge strategy:** `cms-core` provides base messages;
   `apps/<client>/messages/` provides overrides. The deep-merge strategy (locale-by-locale,
   key-by-key) needs implementation in Phase 1 and documentation in `cms-core`'s README.
